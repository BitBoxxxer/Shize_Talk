# Shize Talk — вложения в чат (фото и файлы)

## Что нового

- В чате можно отправить **фото** (сжимается на устройстве перед загрузкой,
  как и аватарки — ресайз по большей стороне до 1600px + JPEG) или
  **любой файл** до 20 МБ.
- Фото показывается превью прямо в пузыре сообщения, тап открывает
  полноэкранный просмотр с зумом. Файл — карточкой с именем/размером,
  тап открывает во внешнем приложении/браузере.
- Хранилище приватное (не как `avatars`) — файл видят только участники
  конкретного чата, доступ проверяется через уже существующую
  `is_chat_participant()`.
- Список чатов теперь показывает "📷 Фото" / "📎 Файл" как превью последнего
  сообщения, если оно без подписи.

## Порядок применения

### База данных
1. Supabase → SQL Editor → выполнить `supabase/12_message_attachments.sql`
   (нужны уже применённые миграции 05, 06, 07 — там определена функция
   `is_chat_participant`, на которую опирается эта миграция).

### Код
1. Заменить `pubspec.yaml` на версию из этого архива (добавлены
   `file_picker: ^11.0.2` и `url_launcher: ^6.3.2`).
2. Заменить `lib/screens/chat_screen.dart` на версию из этого архива.
3. Добавить новый файл `lib/services/chat_attachment_compress.dart`.
4. `flutter pub get`.
5. `flutter run`.

## Технические решения (на будущее, если будешь дебажить)

- **Bucket `chat_attachments` не публичный** (в отличие от `avatars`) —
  доступ выдаётся через `createSignedUrl()` (Storage SDK), ссылка живёт
  сутки. Кэшируется в памяти на время жизни экрана чата
  (`_signedUrlCache`), не сохраняется в БД (истекающие ссылки бессмысленно
  хранить постоянно).
- **Путь в Storage**: `chat_attachments/{chat_id}/{временный_id}.{ext}` —
  первый сегмент пути (`chat_id`) используется в RLS-политиках через
  `storage.foldername(name)[1]`, поэтому доступ проверяется по
  принадлежности к чату, а не по владельцу файла (как у аватарок) — так
  оба участника переписки видят вложение, а не только отправитель.
- **`storage.create_signed_url` не существует как вызываемая SQL-функция**
  (проверено — есть открытый feature request на GitHub Supabase на эту
  тему). Подписанные ссылки создаются только через клиентский SDK, поэтому
  в SQL-миграции для этого нет отдельного RPC — правильно и ожидаемо.
- Сообщение теперь может быть без текста (`content` стал nullable) —
  добавлен constraint `messages_content_or_attachment_check`: должен быть
  либо непустой текст, либо вложение (или оба — подпись к фото).
- Если `insert` в `messages` падает уже после успешной загрузки файла в
  Storage — код подчищает "осиротевший" файл (`_tryDeleteOrphanedAttachment`),
  чтобы не копился мусор в бакете.
