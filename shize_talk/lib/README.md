# Shize Talk — обновление: юзернеймы, друзья, отдельные чаты, ретро-стиль

## 1. Порядок применения

### База данных
1. Supabase → SQL Editor → выполнить `supabase/05_usernames_friends_chats.sql`
   (только после того, как уже выполнены прежние миграции: profiles,
   check_invite_token / redeem_invite / create_invite / list_invites /
   revoke_invite, таблица messages).
2. Затем выполнить `supabase/06_profile_update.sql`.
3. Затем выполнить `supabase/07_fix_chat_participants_recursion.sql`
   (исправляет ошибку "infinite recursion detected in policy for relation
   chat_participants" — см. раздел 7 ниже).
4. Проверить, что каждая миграция отработала без ошибок (SQL Editor должен
   показать "Success. No rows returned" на каждый блок).

### Код
1. Скопировать содержимое `main.dart`, `theme/`, `screens/` в свой Flutter-проект,
   заменив старые файлы.
2. Положить `assets/logo.jpg` (лого, которое вы прислали) в `assets/` проекта.
3. В `pubspec.yaml` добавить:
   ```yaml
   flutter:
     assets:
       - assets/logo.jpg
   ```
4. `flutter pub get`, затем `flutter run`.

## 3. Что изменилось логически

- **Юзернеймы** (`@username`, 3–20 символов, латиница/цифры/`_`, уникальны без
  учёта регистра). Задаются один раз на экране `UsernameSetupScreen` — сразу
  после регистрации по инвайту или при первом входе, если юзернейм ещё не
  задан. Изменить его позже можно тем же RPC `set_username` — теперь есть
  и отдельный экран профиля (см. раздел 4).
- **Друзья**: заявка по юзернейму (`send_friend_request`) → принятие/отклонение
  (`respond_friend_request`) → список друзей (`list_friends`). Чат можно
  начать только с другом, не с произвольным пользователем — это заложено в
  `get_or_create_direct_chat` на уровне БД, а не только в интерфейсе.
- **Чаты**: раньше все сообщения лежали в одной таблице без разделения —
  теперь у `messages` есть `chat_id`, и RLS не даёт прочитать чужой чат, даже
  если знать его id. `ChatsListScreen` — стартовый экран после входа, показывает
  список диалогов с превью последнего сообщения.
- **Оформление**: `theme/app_theme.dart` — тёмный фон, неоновый
  фиолетово-синий-бирюзовый градиент (взят с вашего лого), `RetroBackground`
  оборачивает каждый экран для единообразия.

## 5. Обновление: исправлена отправка сообщений (баг "не отправляются")

Была найдена проблема: старый код очищал поле ввода и полагался только на
Realtime-рассылку, чтобы показать сообщение — если что-то падало (ошибка RLS,
сети и т.п.) или Realtime не был включён для таблицы, сообщение пропадало
без единого следа, даже у самого отправителя.

Теперь `chat_screen.dart`:
- берёт вставленную строку напрямую из ответа на `insert(...).select().single()`
  и сразу показывает её в списке — не дожидаясь Realtime;
- если `insert` падает — показывает конкретную ошибку в SnackBar вместо тишины;
- защищено от дублей (если Realtime всё-таки пришлёт то же сообщение ещё раз).

Также в `05_usernames_friends_chats.sql` добавлена команда:
```sql
alter publication supabase_realtime add table public.messages;
```
без неё подписка `postgres_changes` в приложении не получает новые сообщения
"вживую" у получателя (хотя отправитель теперь видит своё сообщение сразу,
независимо от Realtime).

## 6. Как проверить причину, если что-то опять не отправляется

1. **Realtime включён для таблицы `messages`**: Supabase → Database →
   Publications → в публикации `supabase_realtime` должен быть включён
   переключатель для `messages` (страница "Replication" в новом дашборде
   Supabase теперь про read-replicas/pipelines — нужная настройка называется
   Publications).
2. Если появится SnackBar с текстом ошибки при отправке — присылайте этот
   текст, по нему сразу видно причину.
3. Проверить, что все миграции (`05`, `06`, `07`) выполнились без ошибок.

## 7. Обновление: исправлена ошибка "infinite recursion detected in policy
    for relation chat_participants"

Выполнить `supabase/07_fix_chat_participants_recursion.sql` (после 05 и 06).

Причина: политика `chat_participants_select` проверяла участие в чате,
обращаясь к той же самой таблице `chat_participants` (self-join через `cp2`).
При чтении любой строки Postgres заново применяет ту же политику к этой
строке — и так до бесконечности.

Решение: проверка вынесена в отдельную SECURITY DEFINER функцию
`is_chat_participant(chat_id, user_id)`. Внутри такой функции запрос
выполняется от имени владельца функции (роль с BYPASSRLS), поэтому RLS-политика
на `chat_participants` там не перепроверяется и рекурсии не возникает.
Заодно на эту же функцию переведены политики `chats_select`,
`messages_select` и `messages_insert` — для единообразия.

- Индикатор "прочитано"/статус "онлайн".
- Пуш-уведомления о новых сообщениях, когда приложение свёрнуто.

## 9. Обновление: локальное время, онлайн/был(а) в сети, галочки прочтения, расширенный профиль

1. Выполнить `supabase/08_presence_read_receipts_profile.sql` (после 05, 06, 07).
2. **Локальное время**: `Message.fromMap` теперь делает `.toLocal()` сразу при
   разборе — время сообщений показывается в часовом поясе устройства, а не
   сервера/UTC.
3. **"В сети"/"был(а) в сети"**: каждое устройство раз в 25 секунд, пока открыт
   чат, дёргает `touch_presence()`. Порог "в сети" — последняя отметка не
   старше ~45 секунд. Это упрощённый вариант (без полноценного Presence через
   Realtime-каналы) — задержка показа "не в сети" может быть до ~45 секунд,
   этого достаточно для MVP.
4. **Галочки прочтения**: `chat_participants.last_read_at` — обновляется при
   открытии чата и при получении новых сообщений, пока чат открыт. Своя
   галочка (✓ / ✓✓ синяя) у отправленных сообщений сравнивается с
   `last_read_at` собеседника, обновляется в реальном времени через Realtime
   на `chat_participants`.
5. **Профиль**: добавлены поля "О себе" (до 200 символов) и "Дата рождения" —
   `profile_screen.dart` теперь их редактирует через `update_profile_details`.
   Кнопки "Добавить аккаунт" (заглушка — множественный вход ещё не реализован)
   и "Выйти из аккаунта" перенесены в самый низ экрана, под разделитель.
