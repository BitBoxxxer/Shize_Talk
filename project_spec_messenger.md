# Закрытый мессенджер для семьи и друзей — проектный документ

> Личный проект. Цель: удобное общение с родными/друзьями без риска попадания приложения в открытый доступ. Регистрация — только по приглашению от админа (владельца проекта).

---

## 1. Концепция

- Не публичный мессенджер, а закрытый круг: доступ получают только те, кому лично выдана ссылка-приглашение.
- Никакого "сарафанного радио" — приглашения создаёт **только один человек** (админ/владелец), обычные пользователи такой возможности не имеют.
- После первого входа по инвайт-токену пользователь привязывает аккаунт к телефону/email — это делает его основным способом входа/восстановления, чтобы не терять доступ.

---

## 2. Модель доступа

### Роли
- **admin** — вы. Единственный, кто может создавать инвайт-токены.
- **member** — обычный пользователь, никакого доступа к admin-функциям (проверка роли обязательно на бэкенде, не только скрытие кнопки в UI).

### Поток регистрации
1. Админ создаёт инвайт-токен через отдельное **desktop-приложение**.
2. Ссылка/QR передаётся конкретному человеку лично (не через общедоступные каналы).
3. Пользователь переходит по ссылке → приложение проверяет токен (валиден? не использован? не истёк?).
4. Если ок — экран ввода телефона/email.
5. Отправляется OTP-код на указанный контакт.
6. После подтверждения OTP — аккаунт создаётся, роль `member`, токен помечается использованным (max_uses: 1), привязывается к контакту пользователя.
7. Повторный вход в будущем — уже просто через OTP на телефон/email, новый инвайт не нужен.

### Защита токенов
- `expires_at` — например, 48 часов с момента создания.
- `max_uses: 1`, проверка в транзакции (защита от race condition при параллельном использовании).
- Возможность **revoke** неиспользованных токенов из admin-панели.

---

## 3. Admin-доступ (отдельное desktop-приложение)

- Отдельное Flutter desktop-приложение (Windows/macOS/Linux), не встроено в мобильное.
- Общая бизнес-логика (модели, API-клиент) вынесена в shared Dart-пакет и переиспользуется между mobile и desktop:
  ```
  packages/core        — общие модели, API-клиент, работа с токенами
  apps/mobile          — приложение для всех пользователей
  apps/admin_desktop    — только у вас, admin-функции
  ```
- **Авторизация в admin-приложение — через GitHub OAuth.**
  - Supabase Auth поддерживает GitHub OAuth из коробки.
  - В RLS-политике / серверной проверке сверяется `github_username` с единственным разрешённым логином (вашим).
  - Отдельной системы паролей/2FA поддерживать не нужно — GitHub OAuth сам покрывает эту функцию.
- Функции admin-приложения:
  - Создание нового инвайт-токена (+ генерация QR-кода для передачи вживую).
  - Список активных / использованных / просроченных инвайтов.
  - Revoke (отзыв) неиспользованного токена.
- Репозиторий admin-приложения — **отдельный приватный репозиторий**, не публикуется вместе с mobile-частью.

## 4. Технический стек

- **Клиент:** Flutter/Dart (mobile + desktop admin-app из общей кодовой базы).
- **Backend/BaaS:** Supabase (managed Postgres).
  - Auth: email/phone OTP из коробки + GitHub OAuth для админки.
  - Realtime: подписка на новые сообщения через Postgres publication (задержка ~100–300мс в пределах региона).
  - Storage: S3-совместимое хранилище с CDN — для фото/видео/голосовых.
  - Row Level Security (RLS): обычный пользователь физически не может обратиться к таблице `invites`.
- Flutter SDK: `supabase_flutter` (работает на mobile и desktop).

### Почему Supabase, а не свой бэкенд с нуля
Даёт ~80% инфраструктуры бесплатно (Auth, Realtime, Storage, БД) — не нужно поднимать отдельный Node/Go сервер или разворачивать Matrix/Synapse. Пишете только специфичную логику: инвайты, admin-панель.

## 5. Схема БД (черновик)

```
users:
  id
  phone / email (идентификатор для входа)
  role            -- 'admin' | 'member'
  created_at
  invite_id       -- какой инвайт использовал при регистрации

invites:
  id
  token           -- случайная строка (32 байта, base64url)
  created_by      -- admin_id
  used            -- bool
  used_by         -- nullable, user_id
  expires_at
  max_uses        -- обычно 1
  revoked         -- bool

messages:
  id
  chat_id
  sender_id
  content         -- текст
  created_at

media:
  id
  message_id
  file_url        -- ссылка в Storage
  thumbnail_url
  mime_type
  size
  duration        -- для видео/аудио
```

**Важно:** в Postgres хранятся только текст сообщений и метаданные файлов. Сами файлы — только в Storage. Это критично на бесплатном тарифе (лимит БД обычно ~500MB, лимит Storage отдельный и обычно щедрее).

## 6. Обработка медиа (по принципу YouTube — сжатие + адаптивная выдача)

### Общий принцип
Тяжёлая обработка (особенно видео) — **на устройстве отправителя перед загрузкой**, а не на сервере. Это разгружает бэкенд полностью и не упирается в лимиты бесплатных серверных тарифов (Supabase Edge Functions ограничены по времени выполнения, не годятся для тяжёлых CPU-задач вроде видеокодирования).

| Тип | Кодек/формат | Где обрабатывается | Инструмент во Flutter |
|---|---|---|---|
| Фото | WebP (легче JPEG на 25-35%) + превью низкого разрешения | Клиент | стандартные пакеты сжатия изображений |
| Видео | H.264/H.265, сжатие перед отправкой | Клиент | `video_compress` / `ffmpeg_kit_flutter` |
| Голосовые | Opus (как в Telegram/WhatsApp) | Запись сразу в этом формате | пакет `record` |

### UX-приём для ощущения скорости
Optimistic UI: сообщение показывается в чате сразу (локально), пока идёт фоновая загрузка/обработка — собеседник видит отправку "мгновенно".

## 7. Регион сервера и распределённые пользователи

- Родные могут находиться в разных странах — единый регион БД неизбежно даст кому-то бóльшую задержку.
- **На старте:** выбрать регион, ближе к большинству пользователей (или к себе, как к самому активному admin). Для текстовых сообщений разница в 100-300мс не ощущается пользователем.
- **Медиафайлы** раздаются через встроенный CDN Storage — расстояние до региона БД тут почти не играет роли, отдача идёт с ближайшей ноды.
- **Не усложнять заранее.** Multi-region Postgres (Neon, CockroachDB, платный тариф Supabase) — опция на будущее, если реально почувствуется лаг на практике. Не проектировать это на старте.
- Юридический момент (data residency/GDPR) для закрытого некоммерческого круга людей не является блокером — учитывать только если проект когда-либо выйдет за пределы личного использования.

---

## 8. Ограничения бесплатного тарифа Supabase, которые стоит держать в голове

- Cold start Edge Functions: +1-2 сек на первый вызов после простоя — не критично при нечастых фото/видео.
- Connection pooling: лимит одновременных соединений к Postgres (~60) — с большим запасом для закрытого круга.
- Realtime: лимит одновременных подключений (~200) — тоже с запасом.
- Лимит объёма БД (~500MB на free-тарифе) — не проблема, если в Postgres хранится только текст и метаданные, а не файлы.

## 9. Роадмап (предлагаемая последовательность)

1. **MVP авторизации:** Supabase-проект, таблицы `users`/`invites`, GitHub OAuth для admin-доступа, RLS-политики.
2. **Admin desktop-приложение:** создание/просмотр/revoke инвайтов, генерация QR.
3. **Mobile-приложение:** экран входа по инвайт-токену → OTP → регистрация.
4. **Базовый чат:** текстовые сообщения через Realtime, optimistic UI.
5. **Медиа:** загрузка фото с клиентским сжатием в WebP, затем голосовые (Opus), затем видео (сжатие на клиенте).
6. **Полировка:** статусы прочтения, уведомления (push через FCM), возможно — групповые чаты.
